4. 在上述基础上，补出 `membership / role` 的最小配置入口，让权限模型开始可以直接操作，而不是只有投影没有入口。

## 主要产出

### 1. 统一 task / governance 主链继续收口

今天把前一阶段已经接上的任务主链继续做实：

- `TaskApi` 持续承接 `scan / connect / analyze / snapshot / share-link`
- 审批、通知、事件、审计引用已经能沿同一条链返回
- `admin_api` 和 `webui` 里的关键后台动作已经不再走演示直路，而是回到 `TaskResponse + ArtifactRecord + Event`

这意味着：

- `approval flow` 已经不只是 schema 或底层能力，而是管理面可操作的真实闭环
- `notification contract` 已经不只是 request，会有真实 delivery attempt
- `artifact / share-link / media result contract` 这条原先在文档里的明确待办，已经基本落到主链

### 2. access control 从临时保护推进到平台治理

今天把访问控制做成了“两层门禁”：

- `transport gate` 仍保留本机 / 局域网保护
- `authorization gate` 新增 `workspace / identity / membership / permission` 判定

平台投影侧已经能生成并消费：

- `Workspace`
- `UserAccount`
- `Membership`
- `IdentityBinding`
- `PermissionBinding`

管理 API 里与状态、审批、设备操作相关的入口都开始经过这层授权判断。也就是说，当前系统已经不再只是“谁在本机就能管”，而是平台对象开始真正参与权限决策。

### 3. `AdminConsoleState` 开始从事实源退回兼容层

今天对 `admin_console` 做的最关键调整，不是再加字段，而是改写它和 `platform.*` 的关系：

- 加载时：优先从 `platform.*` 回填 legacy 视图
- 保存时：不再整块重算覆盖 `platform.*`
- 自定义 membership / platform 记录现在可以被保留下来

这一步的意义非常大，因为它把 `AdminConsoleState` 从“平台主配置数据库”往“兼容壳 + 过渡层”推进了一步，更贴近数据模型文档中对它的定位。

### 4. `membership / role` 配置入口落地

在前一条基础上，今天补了成员和角色的最小管理入口：

- `GET /api/access/members`
- `POST /api/access/members/{user_id}/role`

同时，文档 WebUI 已经有对应成员列表和角色修改入口。这里改的是 `platform.memberships` 里的真实角色，而不是单纯改旧投影字段。

当前策略保持克制：

- `owner` 可见但不可在该入口直接改写
- 只开放真实已经存在的角色枚举
- 不新增额外的“成员管理系统”抽象

这一步符合框架，也没有过度设计。
