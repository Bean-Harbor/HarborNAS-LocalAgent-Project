# 日报 — 2026-04-15（北京时间）

## 今日目标

围绕 HarborNAS / HarborOS 家庭智能平台重新收敛整体架构，避免后续在摄像头、小米/萤石、多 AIoT 设备和云端补能能力接入时反复重搭。

## 今日结论

今天把项目的北极星定义正式钉死为：

> 一个以 IM 为统一入口、以设备协同和媒体数据流为核心、以本地优先与云端补能为原则、通过智能编排、数据脱敏与统一账号凭据治理，统一编排家庭 AIoT 设备与 NAS/HarborOS 的本地优先家庭智能平台。

在这个定义基础上，今天完成了三件关键事：

1. 固化整体分层和顶层架构图，统一文档口径。
2. 完成平台级数据模型设计，并在 Rust 中落下第一批 `control_plane` 根对象。
3. 完成设备主链路的第一阶段迁移，让 `registry / hub / discovery / device discovery executor` 开始转向 `snapshot-first`，同时保留 `CameraDevice` 兼容视图。

## 主要产出

### 1. 架构与文档收敛

更新并统一了以下文档：

- `docs/platform-home-agent-hub-unified-architecture.md`
- `docs/home-agent-hub-top-level-design.md`
- `docs/platform-home-agent-hub-data-model.md`
- `.github/skills/harboros-assistant-v2/SKILL.md`

本次明确的关键架构约束：

- `IM` 和 `配置入口 / 管理后台` 是并列北向入口。
- `HarborBeacon` 只负责会话、自然语义整理和交互，不绕过 Runtime 直连云、凭据或设备后端。
- `HarborOS Runtime / Control Plane` 是统一执行与治理中枢。
- `本地优先 / 云端补能` 是执行路径，不是主架构分层。
- 所有云调用都必须经过 `数据脱敏 + 统一凭据治理`。

### 2. 平台级数据模型与 schema 落地

新增/完善了 `src/control_plane/` 下的平台根对象，包括：

- 账号与身份：`users.rs`、`auth.rs`
- 凭据与云端治理：`credentials.rs`
- 设备与数字孪生：`devices.rs`
- 媒体与摄像头能力：`media.rs`
- 任务运行对象：`tasks.rs`
- 事件与自动化：`events.rs`、`automation.rs`
- 审批、审计与模型路由相关结构：`approvals.rs`、`audit.rs`、`models.rs`

这一步的意义是把“平台真相”从单一 `CameraDevice` 抽出来，建立统一落点，为后续 `task / admin / device / provider` 的迁移提供稳定基座。

### 3. 设备主链路批次 A 迁移完成

今天最重要的代码迁移，是把设备主流程从 camera-only 视角推进到平台快照视角：

- `src/runtime/registry.rs`
  - 引入并增强 `DeviceRegistrySnapshot`
  - 支持 legacy `Vec<CameraDevice>` 和新快照格式兼容读取
  - 增加 camera 的增量 upsert 能力
  - 保证 camera 更新不会抹掉非 camera 的平台记录

- `src/runtime/hub.rs`
  - `upsert_devices()` 改成 snapshot-first 路径

- `src/runtime/discovery.rs`
  - discovery 内部先累计到平台快照，再在边界层吐 `connected_devices`

- `src/orchestrator/executors/device_discovery.rs`
  - 执行器内存主态从 `Vec<CameraDevice>` 改为 `DeviceRegistrySnapshot`
  - `discover / list / get / update` 统一先操作 snapshot，再在边界层返回旧视图

这意味着：

- `CameraDevice` 还在，但已经不再是平台唯一事实来源
- `registry` 已经具备承接多品牌、多设备类型的基础形态
- 后续可以继续迁移 `task_api / domain / admin_console`，而不用再次推翻设备层
