# 2026-03-23 联调日志

## 目标

对 HarborOS (http://192.168.3.172/) 进行 **纯 Rust 二进制** 实时联调。

## 关键发现

- REST API (`/api/v2.0/service/*`) 在这台 HarborOS 上返回 **404**，不可用。
- **WebSocket middleware API** (`ws://host/websocket`) 可用且是官方主路径。
- TrueNAS middleware 的 `service.query` filter 参数格式需 **双层数组**: `[[["field","=","value"]], {options}]`。

## 代码变更

1. **`Cargo.toml`**: 新增 `tungstenite = "0.26"` WebSocket 依赖。
2. **`src/orchestrator/executors/harbor_ops.rs`**: 新增 `MiddlewareWsExecutor`。
   - 连接 `ws://host/websocket`，`auth.login` 认证，调 `service.query` / `service.control`。
   - 实现 `Executor` trait，路由为 `MiddlewareApi`（最高优先级）。
3. **`src/main.rs`**: 当 `--harbor-url` + `--harbor-user` + `--harbor-password` 时使用 WS executor；`--harbor-api-key` 时仍走 HTTP REST。

### 飞书 Echo Bot — protobuf WS v2 长连接

**二进制**: `src/bin/feishu_echo_bot.rs`  
**提交**: `f6e2596`

#### 技术难点

| 问题 | 原因 | 解决 |
|------|------|------|
| 9499 Bad Request | Body 用了 snake_case `app_id` | SDK 用 PascalCase `AppID`/`AppSecret` + `locale: zh` header |
| Binary frames 无法解析 | 旧代码假设 JSON text | 飞书 WS v2 用 protobuf，从 SDK 逆向 pbbp2.proto |

#### 逆向 protobuf Schema

```protobuf
message Header { string key=1; string value=2; }
message Frame  { uint64 SeqID=1; uint64 LogID=2; int32 service=3;
                 int32 method=4; repeated Header headers=5;
                 string payload_encoding=6; string payload_type=7;
                 bytes payload=8; string LogIDNew=9; }
```

#### 测试结果

用户在飞书发 "hello" → Bot 收到 protobuf DATA frame → 解析 `im.message.receive_v1` → REST reply API 回复 "hello" → **用户确认收到** ✅
