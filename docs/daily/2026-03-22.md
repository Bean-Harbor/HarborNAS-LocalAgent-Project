## HarborOS 联调指南

### 快速理解

HarborBeacon 是「IM → NAS 操作 → 回复」的中间层。HarborOS 只需关心两个接口：

```
用户 IM ──→ [HarborBeacon 管道] ──→ MCP Tool Call ──→ [HarborOS 执行器] ──→ 结果 ──→ 回复用户
```

### 第二步：消息入口（选择一种）

**方式 A — Long Connection（无需公网 IP，推荐）**

```python
from harborbeacon.long_connection import Gateway, FeishuWsConfig
from harborbeacon.dispatcher import Dispatcher

gateway = Gateway(on_message=dispatcher.handle)
gateway.register_feishu(FeishuWsConfig(app_id="xxx", app_secret="xxx"))
gateway.start_all()
```

**方式 B — Webhook（需要公网 IP 或内网穿透）**

```python

receiver.start()
```

### 第三步：HarborOS 执行器对接

HarborBeacon 调用 HarborOS 的路径：`Dispatcher → McpServerAdapter → Runtime → Router → Executor`

如果你要新增一个 HarborOS 操作，只需实现 Executor 接口：

```python
from orchestrator.contracts import Route, ExecutionResult, StepStatus

class MyHarborOSExecutor:
    @property
    def route(self) -> Route:
        return Route.MIDDLEWARE_API  # 或 HARBOR_CLI

    def is_available(self) -> bool:
        return True  # 检查 HarborOS 是否可达

    def execute(self, action, *, task_id, step_id) -> ExecutionResult:
        # 调用 HarborOS API
        result = call_harbor_os(action.resource)
        return ExecutionResult(
            task_id=task_id,
            step_id=step_id,
            executor_used="harbor_os_api",
            status=StepStatus.SUCCESS,
            result_payload=result,
        )
```

然后注册到 Router：

```python
from orchestrator.router import Router

router = Router([MyHarborOSExecutor()])
```

msg.message_id   # "om_xxx" — 平台消息ID，用于去重和回复

- **会话隔离**：群聊用 chat_id 做 session key，不同用户在同一群共享上下文

### 联调注意事项 Checklist

在与 HarborNAS（github.com/HarborNAS）进行联调时，逐项确认以下内容：

#### ✅ 1. Import 路径（breaking change）

所有导入必须使用新包名，否则会 `ModuleNotFoundError`：

```python
# ❌ 旧（会报错）
from harborclaw.channels import InboundMessage

# ✅ 新
from harborbeacon.channels import InboundMessage
```

#### ✅ 2. 路由优先级不可绕过

路由固定顺序：`middleware_api → midcli → browser → mcp`

HarborOS 端的 Executor **必须先实现 `middleware_api` 路由**，不能跳过直接走 `midcli`。`route` 属性返回值决定 Router 选哪条路。

#### ✅ 3. 高风险操作有强制审批门，联调时别忽略 BLOCKED 状态

`RiskLevel.HIGH` / `CRITICAL` 的 Action 会被 `PolicyGuard` 自动拦截（`StepStatus.BLOCKED`），**不会执行**。如果联调时某操作没反应，先检查：

```python
action.risk_level  # 是否是 HIGH/CRITICAL？
action.requires_approval  # 是否被自动设为 True？
```

#### ✅ 4. Action 合约字段必须完整

```python
Action(
    domain="system.harbor_ops",    # ✅ 必填
    operation="files.batch_ops",   # ✅ 必填
    resource={"path": "/data"},    # ✅ 必填，dict 类型
    risk_level=RiskLevel.LOW,      # ✅ 必填
    # requires_approval 由 risk_level 自动推断，无需手动设置
)
```

缺少任意必填字段会在 `Action.__post_init__` 阶段抛 `TypeError`。

#### ✅ 5. IM 接入方式选择

| 场景 | 推荐方式 |
|------|---------|
| NAS 在内网、无公网 IP | Long Connection（`Gateway`） |
| 有公网 IP 或内网穿透 | Webhook（`WebhookReceiver`） |

两种方式**不能同时启动**同一个 bot（会收到重复事件）。

#### ✅ 6. 群聊联调：记得 @ 机器人或用问句触发

`should_respond_in_group()` 对普通聊天消息**静默过滤**，只有以下情况才会响应：
- 消息中包含 `@机器人`（`mentions` 列表非空）
- 消息以 `?` / `？` 结尾
- 消息包含请求动词：`帮`、`请`、`能否`、`可以`、`如何`

联调时如果消息发出去没反应，先 print `should_respond_in_group()` 的返回值排查。

#### ✅ 7. 消息去重：避免重复 message_id

`MessageDedup` 在 TTL 内会丢弃相同 `message_id` 的消息。联调测试时如果同一条消息"消失"，检查是否无意间复用了相同的 `message_id`（常见于 mock 测试数据）。
