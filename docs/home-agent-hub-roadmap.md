# Home Agent Hub Roadmap

> 当前定位说明（2026-04-13）  
> 本文档是当前正在执行的产品路线，属于 HarborNAS 平台主干下的首个垂直域。  
> 长期平台骨干仍由 `HarborNAS-LocalAgent-Roadmap.md` 定义；Home Agent Hub 不再作为旁路 demo，而是要通过统一 Task API / Runtime / Audit 入轨平台。

更新时间：2026-04-10

## 1. 当前产品锚点

当前版本不追求一次性完成完整平台，而是优先做出一条稳定、可演示、可复用的黄金链路：

`局域网自动扫描 -> 识别可用 RTSP 摄像头 -> 抓帧 -> AI 检测 -> 截图推送到 IM`

这条链路将作为 Home Agent Hub 的 MVP。

同时，这条 MVP 链路承担额外职责：

- 作为 HarborNAS 平台首个垂直域的真实落地样板
- 反向推动 Task API、artifact contract、event contract、补参机制等平台接口收敛

如果这条链路跑通，说明以下核心能力都已经建立：

- 局域网设备发现
- 统一设备模型
- 流媒体输入与截图输出
- AI provider 接入
- 事件触发与任务执行
- IM 通知闭环
- WebUI / 管理后台的最小治理基础

---

## 2. MVP 定义

## 2.1 用户故事

管理员在后台完成摄像头接入和 AI 检测配置后，普通用户可以：

1. 系统自动扫描局域网中的摄像头候选设备
2. 系统自动识别可用的 RTSP 摄像头流
3. 在系统中看到已发现并已接入的摄像头
4. 触发摄像头实时拉流或抓取截图
5. 让系统对画面做一次 AI 检测
6. 在检测命中后，把截图和文本说明推送到指定 IM 渠道

## 2.2 MVP 范围

包含：

- 局域网自动扫描候选摄像头设备
- 基于 `ONVIF / SSDP / mDNS / RTSP Probe` 的候选发现
- 自动识别并验证可用 RTSP 流
- RTSP 拉流或单帧抓取
- 最小设备注册中心
- 单路 AI 检测调用
- 单条通知工作流
- 飞书优先的 IM 推送
- 前台摄像头查看基础页面
- 后台摄像头接入与工作流配置基础页面

不包含：

- 多协议大规模设备矩阵
- 完整拖拽工作流编辑器
- 完整插件市场
- 多模型自动调度
- 复杂多租户站点治理

## 3. 版本路线

## Milestone 0: Architecture Freeze

目标：
- 冻结 MVP 的领域模型、协议栈和模块边界

交付：
- 顶层设计文档
- WebUI 信息架构
- MVP roadmap
- Phase backlog

完成标志：
- 团队对 MVP 范围和链路一致
- 后续实现不再反复改目标

## Milestone 1: Device Discovery Alpha

目标：
- 能自动扫描局域网中的候选摄像头设备并写入统一设备模型

交付：
- `device.discover` 契约
- 局域网候选设备发现模块
- 设备注册中心 schema
- 后台设备中心最小列表页

完成标志：
- 至少能自动识别 1 路可用 RTSP 摄像头
- WebUI 可看到设备名称、地址、发现方式、在线状态

## Milestone 2: Stream & Snapshot Alpha

目标：
- 能从自动识别出的 RTSP 设备拉取流信息并抓取截图

交付：
- RTSP 连接元数据
- 截图导出能力
- 前台摄像头预览页
- 基础媒体存储路径

完成标志：
- 管理员可在后台测试抓帧
- 普通用户可在前台查看最新截图

## Milestone 3: AI Detection Alpha

目标：
- 能对抓取帧调用一次 AI 检测

交付：
- AI provider 配置
- 检测任务 schema
- 单次检测结果对象
- 检测日志与错误回显

完成标志：
- 输入截图可返回文本化检测结果
- 后台可看到调用状态、耗时、结果摘要

## Milestone 4: IM Notification Beta

目标：
- 检测命中后能把截图和说明推送到 IM

交付：
- 飞书通知模板
- 命中条件配置
- 事件到消息的任务链路
- 审计记录

完成标志：
- 从摄像头截图到飞书收图全链路成功
- 能看到完整执行日志

## Milestone 5: MVP Demo

目标：
- 完成一个可连续演示的 MVP

演示流程：

1. 系统自动扫描局域网并识别摄像头
2. 后台确认并启用该摄像头
3. 后台配置“检测并通知”任务
4. 用户在前台或 IM 触发检测
5. 系统抓图、调用 AI、推送结果到 IM

完成标志：
- 一键演示成功率可接受
- 日志、审计、基础权限完整

## 4. Phase 路线

## Phase 1: 自动发现与接入底座

目标：
- 解决“能不能自动扫描并识别可用 RTSP 摄像头”

重点：
- ONVIF / SSDP / mDNS 候选发现
- RTSP Probe 验证
- Device Registry
- 设备状态模型
- 后台设备页基础版

## Phase 2: 媒体输入与抓图

目标：
- 解决“能不能拿到画面”

重点：
- RTSP 元数据
- ffmpeg/gstreamer 抓帧
- 快照入库
- 前台摄像头页基础版

## Phase 3: AI 检测接入

目标：
- 解决“能不能识别画面”

重点：
- AI provider 网关
- 检测任务对象
- 检测结果对象
- 管理后台模型配置页基础版

## Phase 4: 通知闭环

目标：
- 解决“能不能把结果送出去”

重点：
- 飞书通知
- 消息模板
- 审计记录
- 命中条件配置

## Phase 5: MVP 治理补齐

目标：
- 让 MVP 不只是 demo，而是可管理

重点：
- 最小多用户角色
- 权限边界
- 运行监控
- 错误日志
- 基础系统设置

---

## 5. MVP 成功标准

满足以下条件即可认定 MVP 完成：

1. 系统可发现并注册至少一台摄像头
2. 发现过程无需手工填写摄像头 IP
3. 系统可抓取该摄像头的截图
4. 系统可对截图发起一次 AI 检测
5. 检测结果可连同截图推送到飞书
6. 管理员可在 WebUI 看到设备、检测任务、执行结果
7. 普通用户可在前台查看摄像头结果或通过 IM 收到结果

---

## 6. MVP 之后的扩展顺序

MVP 结束后，建议按以下顺序扩展：

1. PTZ 控制
2. 持续拉流检测
3. 条件分支工作流
4. 多摄像头与多房间
5. 更多 IM 渠道
6. 拖拽式 Workflow Studio
7. Matter / Light / Sensor 联动
8. HarborOS 深度媒体归档
